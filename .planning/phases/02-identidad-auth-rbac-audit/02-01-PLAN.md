## Objetivo

Levantar toda la infraestructura de datos de Nexo en Postgres: schema `nexo`
con las 8 tablas acordadas en `docs/AUTH_MODEL.md`, paquete `nexo/` con
`engine_nexo`, modelos ORM y los servicios de auth **aún sin cablear al
middleware**. Dejar creado un `propietario` inicial y los departamentos
seed, de forma que los siguientes plans (auth, RBAC, audit) ya puedan hacer
login contra datos reales sin tocar DDL.

**Lo que NO entra** (se cubre en plans posteriores):
- Middleware de auth / audit (02-02 y 02-04).
- Endpoints `/login`, `/logout`, `/cambiar-password` (02-02).
- `require_permission` aplicado a routers (02-03).
- UI de `/ajustes/usuarios`, `/ajustes/auditoria` (02-04).

## Decisiones técnicas heredadas del research

- **Research §Standard Stack**: `argon2-cffi 25.1.0`, `itsdangerous 2.2.0`, `slowapi 0.1.9`. Versiones pineadas exactas. Verificadas en PyPI.
- **Research §Pattern 1 (Argon2id)**: parámetros `time_cost=3, memory_cost=65536, parallelism=4, hash_len=32, salt_len=16`. Coinciden con `argon2.profiles.RFC_9106_LOW_MEMORY`.
- **Research §Pattern 6 (engine_nexo)**: `pool_size=5, max_overflow=5, pool_timeout=10, pool_recycle=1800, pool_pre_ping=True`. Coexiste con `engine_app` de SQL Server sin conflicto.
- **Research §Pattern 7 (modelos ORM)**: todas las tablas con `__table_args__ = {"schema": "nexo"}`. `DateTime(timezone=True)` en todas las columnas temporales.
- **Research §Pattern 8 + Pitfall 4**: no usar `REVOKE UPDATE, DELETE` (ambiguo cuando el rol es owner). En su lugar, `GRANT SELECT, INSERT ON nexo.audit_log TO <app_role>` partiendo de un rol que no tiene privilegios previos. **Afinado respecto a `docs/AUTH_MODEL.md` por research #7** (AUTH_MODEL.md dice literalmente "REVOKE UPDATE, DELETE"; research detecta que si `pg_user` es owner, el REVOKE no surte efecto; adoptamos la alternativa del research).
- **Research §Stack Decision 2**: `PERMISSION_MAP` como dict en código (fuente de verdad). La tabla `nexo.permissions` queda como catálogo seed; Phase 5 puede consultarla para construir el sidebar sin imports cruzados.
- **Research §Stack Decision 3**: crear sólo `nexo/db/` y `nexo/services/auth.py`. No crear `nexo/data/repositories/*` — Phase 3.
- **Research §Open Question 2**: se mantiene el rol actual (`settings.pg_user`, hoy `"oee"`) como rol de la app. El script de init **no** crea un rol dedicado `nexo_app`; documenta la limitación en el propio script para Mark-IV. El GRANT-only sobre `audit_log` sigue siendo efectivo siempre que el rol no sea propietario de la tabla — ver tarea 1.4.

### Tarea 1.2 — Paquete `nexo/db/` con engine y modelos ORM

**REQ-ID**: IDENT-02, IDENT-03, IDENT-06.

**Archivos a crear**:
- `nexo/__init__.py` (vacío)
- `nexo/db/__init__.py` (vacío)
- `nexo/db/engine.py`
- `nexo/db/models.py`
- `nexo/services/__init__.py` (vacío, placeholder para tarea 1.3)

**`nexo/db/engine.py` — contenido**:
- Crea `engine_nexo` con URL `postgresql+psycopg2://{pg_user}:{pg_password}@{pg_host}:{pg_port}/{pg_db}` construida desde `api.config.settings`.
- Parámetros de pool (research §Pattern 6): `pool_size=5`, `max_overflow=5`, `pool_timeout=10`, `pool_recycle=1800`, `pool_pre_ping=True`, `echo=settings.debug`.
- `SessionLocalNexo = sessionmaker(bind=engine_nexo, expire_on_commit=False)`.
- NO importar `api/database.py` ni tocar `engine` de SQL Server.

**`nexo/db/models.py` — contenido** (research §Pattern 7):
- `class NexoBase(DeclarativeBase): pass`.
- Tabla asociación `user_departments` (`schema="nexo"`).
- `NexoUser` (`nexo.users`): `id, email (unique), password_hash, role (20), active, must_change_password (default True), last_login (nullable), created_at`. Relationships: `departments` (N:M), `sessions` (1:N cascade all, delete-orphan).
- `NexoRole` (`nexo.roles`): `id, code (unique: propietario/directivo/usuario), name`.
- `NexoDepartment` (`nexo.departments`): `id, code (unique: rrhh/comercial/ingenieria/produccion/gerencia), name`.
- `NexoPermission` (`nexo.permissions`): `id, module, action, role_code, department_code (nullable)`. Catálogo seed.
- `NexoSession` (`nexo.sessions`): `id, user_id (FK), token (unique, index), expires_at, created_at`.
- `NexoLoginAttempt` (`nexo.login_attempts`): `id, email (index), ip, failed_at (index)`.
- `NexoAuditLog` (`nexo.audit_log`): `id, ts (index), user_id (FK nullable), ip, method, path, status, details_json (Text nullable)`.

**Regla de estilo**:
- Usar `datetime.now(timezone.utc)` en todos los defaults, **no** `datetime.utcnow()` (research §Pitfall 3, deprecated en Python 3.12).
- Usar `DateTime(timezone=True)` en todas las columnas temporales.
- `password_hash` con `String(200)` (Argon2id PHC string cabe en <150 chars, margen amplio).

**Criterio de hecho**:
- `docker compose exec web python -c "from nexo.db.engine import engine_nexo; print(engine_nexo.url)"` imprime la URL construida.
- `docker compose exec web python -c "from nexo.db.models import NexoUser, NexoAuditLog; print(NexoUser.__tablename__, NexoAuditLog.__table_args__)"` imprime `users {'schema': 'nexo'}`.
- `make dev` sigue arrancando (no importamos `nexo/` desde `api/main.py` todavía).

**Acciones**:
- Añadir al `Settings`:

  session_cookie_name: str = Field("nexo_session", validation_alias=AliasChoices("NEXO_SESSION_COOKIE_NAME"))

  ```

  # NEXO_SESSION_TTL_HOURS=12
  # NEXO_PG_HOST=db
  # NEXO_PG_PORT=5432
  ```
- **Antes de reiniciar**: operador genera y pega en `.env` (fuera de tracking) la `NEXO_SECRET_KEY` real. El ejecutor **no** genera ni escribe la clave en `.env` directamente (denyList del entorno).

**Criterio de hecho**:
- `docker compose exec web python -c "from api.config import settings; print(len(settings.secret_key))"` imprime un entero > 40 (clave razonable).
- Arrancar la app con `NEXO_SECRET_KEY` vacío en `.env` falla con `ValidationError` claro.

### Tarea 1.4 — `scripts/init_nexo_schema.py` idempotente

**REQ-ID**: IDENT-02, IDENT-03, IDENT-06.

**Archivos**:
- `scripts/init_nexo_schema.py` (crear)

**Acciones** (research §Pattern 8 + §Pitfall 5):

1. `CREATE SCHEMA IF NOT EXISTS nexo` en `engine_nexo.begin()` **antes** de `create_all`.
2. `NexoBase.metadata.create_all(engine_nexo)` — crea las 8 tablas.
3. Seed idempotente de roles:
   - `INSERT ... ON CONFLICT (code) DO NOTHING` para 3 filas: `propietario`, `directivo`, `usuario`.
4. Seed idempotente de departments:
   - 5 filas: `rrhh` (Recursos Humanos), `comercial` (Comercial), `ingenieria` (Ingeniería), `produccion` (Producción), `gerencia` (Gerencia).
5. Seed idempotente de `nexo.permissions` — snapshot del `PERMISSION_MAP` de research §Pattern 4. Una fila por tupla `(module, action, role, department)`. Catálogo informativo; la auth real usa el dict en código (decision 02-03).
6. Privilegios audit log (research §Pattern 8, alternativa de GRANT-only):
   ```sql
   -- Ejecutar como owner del schema. Asume que el rol app (settings.pg_user)
   -- NO es el owner del schema/tabla. En dev el schema se crea desde el
   -- script que corre como el propio pg_user; en ese caso el REVOKE que
   -- sigue es explícito por si acaso.
   REVOKE ALL ON nexo.audit_log FROM PUBLIC;
   GRANT SELECT, INSERT ON nexo.audit_log TO <pg_user>;
   GRANT USAGE, SELECT ON SEQUENCE nexo.audit_log_id_seq TO <pg_user>;
   ```
   **Nota explícita dentro del script** (como comentario y como log de arranque):
   > Si el rol Postgres que corre este script es owner de `nexo.audit_log`, el
   > GRANT-only no es suficiente para impedir UPDATE/DELETE (owner siempre los
   > tiene). El test IDENT-06 lo detectará. En Mark-IV se introduce un rol
   > `nexo_app` dedicado. En Mark-III, en LAN sin acceso directo a psql, el
   > riesgo es aceptable.
7. El script **no crea usuarios**. Delega en `scripts/create_propietario.py` (tarea 1.5).
8. Idempotencia: correr el script dos veces seguidas debe terminar sin error y con el mismo estado.

**Modo de ejecución**:
```bash
docker compose exec web python scripts/init_nexo_schema.py
```

**Criterio de hecho**:
- Al terminar, `docker compose exec db psql -U $NEXO_PG_USER -d $NEXO_PG_DB -c "\dt nexo.*"` lista las 8 tablas (`users, roles, departments, user_departments, permissions, sessions, login_attempts, audit_log`).
- `SELECT code FROM nexo.roles;` devuelve 3 filas.
- `SELECT code FROM nexo.departments;` devuelve 5 filas.
- `SELECT COUNT(*) FROM nexo.permissions;` devuelve > 10 filas.
- Correr el script una segunda vez termina con exit 0 y sin duplicar filas.

### Tarea 1.5 — `scripts/create_propietario.py` interactivo

**REQ-ID**: soporte para IDENT-01 / IDENT-02 (habilita el primer login).

**Archivos**:
- `scripts/create_propietario.py` (crear)

**Acciones** (research §Pattern 11):

- `getpass.getpass` para password sin echo.
- Idempotencia: si ya existe un usuario con `role="propietario"`, imprime "Ya existe un propietario" y sale con exit 0 (no lanza excepción — correr el script dos veces no es un error).
- Validaciones en el propio script (antes de tocar DB):
  - Email no vacío.
  - Password igual en primer y segundo prompt.
  - Password ≥ 12 caracteres (decisión LOCKED en AUTH_MODEL.md §Passwords).
- Al crear el usuario:
  - `role="propietario"`.
  - `active=True`.
  - `must_change_password=False` — el propietario inicial bootstrap **no** pasa por el flujo de cambio obligatorio (si no, caería en bucle redirect porque todavía no existe la página `/cambiar-password` cuando se crea el usuario).
- No asigna departamentos (el propietario ignora departamento, research §Pattern 4 bypass).

**Criterio de hecho**:
- `docker compose exec web python scripts/create_propietario.py` con input `admin@ecsmobility.com` + password ≥ 12 chars crea el usuario.
- `SELECT email, role, must_change_password FROM nexo.users;` muestra la fila creada.
- Correr el script una segunda vez dice "Ya existe un propietario" y sale con exit 0 sin crear otro.
