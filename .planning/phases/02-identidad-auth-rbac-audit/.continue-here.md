## Critical Anti-Patterns (do NOT repeat these)

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|----------------------|
| Copiar `.env` a la imagen Docker | Durante Plan 02-01 el servicio web fallaba con ValidationError al no recibir `NEXO_SECRET_KEY` porque Compose carga `.env` sólo para interpolación YAML, no lo inyecta al container. Se resolvió con `env_file: - .env` en compose | advisory | Nunca añadir `.env` al Dockerfile COPY. Siempre usar `env_file:` en el servicio del compose. |
| Hardcodear `oee` en Makefile psql targets | El Makefile original asumía user/db hardcoded; el `.env` del operador usa compat OEE_* y el grep `^NEXO_PG_USER=` devolvía vacío | advisory | Targets psql usan `docker compose exec db bash -c 'psql -U $$POSTGRES_USER ...'` para leer del container, no del `.env` del host. |
| Omitir `COPY nexo/` en Dockerfile | `docker compose exec web python scripts/...` fallaba con "No such file" — el Dockerfile no copiaba ni `nexo/` ni `scripts/` | advisory | Cualquier paquete nuevo en raíz se añade al Dockerfile en la sección "Código fuente" antes del `docker compose build`. |
| Asumir que el rol app no es owner del schema Postgres | Todas las tablas creadas por `init_nexo_schema.py` tienen `Owner=oee` (el mismo rol que usa la app). GRANT-only no bloquea UPDATE/DELETE al owner | blocking | Plan 02-04 gate IDENT-06 debe correr DELETE desde el rol app y esperar `permission denied`. Si pasa, aplicar Opción A (rol `nexo_app` dedicado) u Opción B (diferir Mark-IV, documentar en SECURITY_AUDIT.md). No cerrar Phase 2 sin decisión explícita. |
