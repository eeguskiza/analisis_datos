**Phase**: 2 of 7 — Identidad (auth + RBAC + audit)

---

## Objetivo

Cerrar toda la capa de **sesión**: middleware de auth que corre antes del resto
y redirige / devuelve 401 según el tipo de request; endpoints `/login`,
`/logout` y `/cambiar-password` funcionales; bloqueo progresivo `(user, IP)`
escribiendo en `nexo.login_attempts`; rate limit `slowapi` sobre `/login`;
primer login forzando cambio de password.

Al terminar este plan, un navegador sin cookie es redirigido a `/login`, un
cliente API sin cookie recibe 401, y un usuario autenticado puede navegar la
app. **Todavía sin RBAC por rol/departamento** (eso es 02-03) y sin audit
middleware escribiendo (eso es 02-04).

## Decisiones técnicas heredadas del research

- **Research §Pattern 2**: `URLSafeTimedSerializer` con `salt="nexo-session"`. Cookie firma sólo el `token` (no `user_id`) — la DB es la fuente de verdad para el mapeo `token → user`.
- **Research §Pattern 3 (AuthMiddleware)**: `_PUBLIC_PATHS = {"/login", "/api/health"}`; `_PUBLIC_PREFIXES = ("/static/", "/favicon.ico")`. `/api/health` queda público — decisión del planner según research §Open Question 1: sin esto, los healthchecks de compose y Caddy se rompen.
- **Research §Pattern 3 + §Pitfall 1 (CRÍTICO)**: `app.add_middleware(AuditMiddleware)` PRIMERO, `app.add_middleware(AuthMiddleware)` SEGUNDO. Starlette usa LIFO: el último registrado es el primero en ejecutar. **Auth debe ejecutar primero para poblar `request.state.user` antes de audit.** Tarea 2.3 tiene gate específico.
- **Research §Pattern 9 (slowapi)**: `Limiter(key_func=get_remote_address)` + `@limiter.limit("20/minute")` en `/login`. Registrar `app.state.limiter` y `_rate_limit_exceeded_handler` en `api/main.py`.
- **Research §Pattern 9 (bloqueo progresivo)**: `LOCKOUT_THRESHOLD=5`, `LOCKOUT_MINUTES=15`, tupla `(email, ip)`. Purga al login exitoso.
- **Research §Pitfall 6 (cookie SameSite)**: `SameSite=Lax` (no `Strict`). Las navegaciones HTMX top-level quedan autorizadas.
- **Research §Pitfall 2**: FastAPI 0.135.3 (ya en requirements) cachea `request.body()`; lectura doble es segura. Relevante en 02-04 más que aquí.
- **Research §Pattern 10**: helper `render(template, request, extra)` en `api/deps.py` para inyectar `current_user` sin repetirlo en cada endpoint.
- **Research §Pitfall 3**: `datetime.now(timezone.utc)` en todo el código de auth, nunca `datetime.utcnow()`.

## Tareas

### Tarea 2.1 — `api/middleware/auth.py` + `api/middleware/__init__.py`

**REQ-ID**: IDENT-04, IDENT-09.

**Archivos a crear**:
- `api/middleware/__init__.py` (vacío)
- `api/middleware/auth.py`

**Acciones** (research §Pattern 3):

- Clase `AuthMiddleware(BaseHTTPMiddleware)` con método `dispatch`:
  1. Whitelist:
     - `_PUBLIC_PATHS = {"/login", "/api/health"}` → `return await call_next(request)`.
     - `_PUBLIC_PREFIXES = ("/static/", "/favicon.ico")` → idem.
  2. Lee `cookie = request.cookies.get(settings.session_cookie_name)`.
  3. Si no hay cookie → decide por path:
     - `path.startswith("/api/")` → `JSONResponse({"detail": "Not authenticated"}, status_code=401)`.
     - Resto → `RedirectResponse("/login", status_code=302)`.
  4. Si hay cookie, llama `get_session(cookie, settings.secret_key, db)` con `SessionLocalNexo`. Si devuelve `None` o `user.active is False` → mismo tratamiento que cookie ausente.
  5. Si `user.must_change_password is True` y `path != "/cambiar-password"` → `RedirectResponse("/cambiar-password", status_code=302)`.
  6. Si todo OK: `request.state.user = user` y `return await call_next(request)`.
- Importar `SessionLocalNexo` desde `nexo.db.engine`, y `get_session` desde `nexo.services.auth`.

**Criterio de hecho**:
- Con cookie válida, `request.state.user` queda establecido dentro de los handlers.
- Endpoint HTML sin cookie → redirect 302 a `/login`.
- Endpoint `/api/*` (≠ `/api/health`) sin cookie → 401 JSON.
- `/api/health` sin cookie → 200 OK (whitelisted).
- Usuario con `must_change_password=True` visitando `/` → redirect a `/cambiar-password`.

### Tarea 2.2 — `api/routers/auth.py` con login, logout, cambio de password

**REQ-ID**: IDENT-01, IDENT-09.

**Archivos a crear**:
- `api/routers/auth.py`
- `templates/login.html`
- `templates/cambiar_password.html`

**Acciones** (research §Pattern 2, §Pattern 9):

**Endpoints**:

1. `GET /login` — renderiza `templates/login.html` con form `email + password`. No requiere auth (whitelisted en AuthMiddleware).

2. `POST /login` — decorado con `@limiter.limit("20/minute")`:
   - Parsea form (`email`, `password`).
   - IP = `get_remote_address(request)` (si detrás de Caddy, Caddy inyecta `X-Forwarded-For`; slowapi lo lee si se configura; en Mark-III aceptamos que `request.client.host` es la IP del reverse proxy en prod — no bloqueante para Sprint 1).
   - Paso 1: `check_lockout(email, ip, db)` → si True, renderiza `login.html` con error "Cuenta temporalmente bloqueada. Reintenta en 15 minutos." y HTTP 429.
   - Paso 2: query `NexoUser` por email. Si no existe o `active=False` → `record_failed_attempt(email, ip, db)` y devolver error genérico "Credenciales inválidas" con 401 (no distinguir "usuario no existe" vs "password mala" — evita enumeración de usuarios).
   - Paso 3: `verify_password(user.password_hash, password)` → si False → `record_failed_attempt` + mismo error 401.
   - Paso 4 (éxito):
     - `clear_attempts(email, ip, db)`.
     - `user.last_login = datetime.now(timezone.utc)`; commit.
     - `signed = create_session(user.id, settings.secret_key, db)` — devuelve cookie firmada, crea fila en `nexo.sessions`.
     - `RedirectResponse("/", 303)` + `response.set_cookie(settings.session_cookie_name, signed, httponly=True, secure=True, samesite="lax", max_age=settings.session_ttl_hours*3600)`.
     - Si `must_change_password=True` → redirect a `/cambiar-password` en su lugar (ya lo hace el middleware en la siguiente request, pero cortocircuitar aquí es igualmente válido).

3. `POST /logout`:
   - Lee cookie del request. Si existe, `revoke_session(cookie, settings.secret_key, db)`.
   - `RedirectResponse("/login", 303)` + `response.delete_cookie(settings.session_cookie_name)`.

4. `GET /cambiar-password`:
   - Requiere sesión válida (el middleware ya garantiza `request.state.user`).
   - Renderiza `templates/cambiar_password.html`.

5. `POST /cambiar-password`:
   - Parsea `password_actual, password_nuevo, password_repetir`.
   - Verifica `password_actual` con `verify_password`.
   - Valida `password_nuevo == password_repetir` y `len(password_nuevo) >= 12`.
   - `user.password_hash = hash_password(password_nuevo)`, `user.must_change_password = False`, commit.
   - Revoca sesiones existentes del usuario (`DELETE FROM nexo.sessions WHERE user_id = :uid`) para forzar re-login — patrón estándar de cambio de password.
   - `RedirectResponse("/login", 303)` con flash "Password actualizado. Vuelve a iniciar sesión."

**`templates/login.html`**:
- Formulario minimal en Alpine/Tailwind consistente con el resto de templates.
- Campos: `email`, `password`. Botón "Iniciar sesión".
- Si viene con `?error=...` o mensaje flash server-side, mostrar banner rojo.

**`templates/cambiar_password.html`**:
- Formulario con 3 campos: actual, nuevo, repetir.
- Mensaje claro "Tu cuenta requiere cambiar la contraseña antes de continuar" (si `must_change_password=True`).

**Registro del router en `api/main.py`**:
```python
from api.routers import auth as auth_router  # noqa
# ... después del resto de include_router:
app.include_router(auth_router.router)   # sin prefix /api — es HTML + endpoints de auth
```

**Criterio de hecho**:
- `curl -i http://localhost:8000/login` devuelve 200 con HTML del formulario.
- `curl -X POST http://localhost:8000/login -d "email=X&password=Y"` con credenciales del propietario de 02-01 devuelve 303 + `Set-Cookie: nexo_session=...`.
- Con la cookie, `curl -b "nexo_session=..." http://localhost:8000/api/health` devuelve 200 (y también lo haría sin cookie porque health es público).
- `POST /logout` borra la cookie y la fila en `nexo.sessions`.
- Tras cambiar password, la fila correspondiente en `nexo.sessions` se borra y la app pide re-login.

  ```python
  from slowapi import Limiter, _rate_limit_exceeded_handler
  ```

**Acciones** (research §Pattern 10):

En `api/deps.py`:
```python
from fastapi import Request
from fastapi.responses import HTMLResponse

def render(template_name: str, request: Request, extra: dict | None = None) -> HTMLResponse:
    ctx = {"request": request, "current_user": getattr(request.state, "user", None)}
    if extra:
        ctx.update(extra)
    return templates.TemplateResponse(template_name, ctx)
```
