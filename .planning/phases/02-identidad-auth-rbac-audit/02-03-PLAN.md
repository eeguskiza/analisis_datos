## Objetivo

Convertir la "sesión autenticada" que queda tras 02-02 en "permiso por rol y
departamento". Introducir `require_permission("modulo:accion")` como
dependency factory FastAPI y aplicarla a los 15 módulos de routers
existentes (14 de la lista de CONTEXT.md + `plantillas.py` que también existe
en el repo). El `PERMISSION_MAP` queda como **única fuente de verdad** de los
permisos en código; la tabla `nexo.permissions` sigue siendo catálogo seed
sin consultarse en runtime.

Al terminar:
- `propietario` sigue pasando todos los endpoints (bypass).
- `directivo` / `usuario` sólo pasan si su intersección de departamentos con
  el mapa del permiso es no vacía.
- Un `usuario` de "rrhh" que intenta `POST /api/pipeline/run` recibe 403.

## Decisiones técnicas heredadas del research

- **Research §Pattern 4**: factory `require_permission(permission)` devuelve una async dependency que lee `request.state.user`, hace bypass si `role == "propietario"`, y si no cruza `PERMISSION_MAP.get(permission)` con `user.departments`. Devuelve el `User` para que el handler pueda usarlo; lanza 401 si falta sesión (el middleware ya lo cubre pero es defensivo) y 403 si no hay intersección.
- **Research §Pattern 4 `PERMISSION_MAP` inicial**: es la fuente autoritativa. Copia literalmente este dict en `nexo/services/auth.py` y documenta desviaciones en el PLAN si las hay.
- **Research §Stack Decision 2**: el dict en código gana. `nexo.permissions` está sólo como catálogo seed para Phase 5.
- **Research §Pitfall 7**: el retrofit es mecánico en 15 archivos; fácil olvidar uno. La tarea 3.3 incluye un test de integración que verifica que todos los `/api/*` que tocamos retornan 401/403 según corresponda.
- **Research §Open Question 1**: `/api/health` queda sin `require_permission` — ya whitelisted en `AuthMiddleware` (02-02). El router `health` no recibe retrofit.
- **Research §Anti-Patterns**: NO incluir `propietario` como entrada del `PERMISSION_MAP`. El bypass es por rol, antes del lookup.

## Tareas

### Tarea 3.1 — `PERMISSION_MAP` + `require_permission` en `nexo/services/auth.py`

**REQ-ID**: IDENT-03, IDENT-05.

**Archivos**:
- `nexo/services/auth.py` (modificar, añadir al final)

**Acciones** (research §Pattern 4):

```python
from fastapi import HTTPException, Request, status

# Claves "modulo:accion" → lista de departamentos que tienen acceso.
# propietario ignora este mapa (bypass antes de consultarlo).
# Fuente: research §Pattern 4 + CONTEXT.md + AUTH_MODEL.md §Departamentos.
PERMISSION_MAP: dict[str, list[str]] = {
    # Módulos de producción / planta
    "pipeline:run":       ["ingenieria", "produccion"],
    "pipeline:read":      ["ingenieria", "produccion", "gerencia"],
    "recursos:read":      ["ingenieria", "produccion"],
    "recursos:edit":      ["ingenieria"],
    "ciclos:read":        ["ingenieria"],
    "ciclos:edit":        ["ingenieria"],
    "centro_mando:read":  ["produccion", "ingenieria", "gerencia"],
    "luk4:read":          ["produccion", "ingenieria", "gerencia"],
    "capacidad:read":     ["comercial", "ingenieria", "produccion", "gerencia"],
    "historial:read":     ["ingenieria", "produccion", "comercial", "gerencia", "rrhh"],
    "informes:read":      ["ingenieria", "produccion", "comercial", "gerencia", "rrhh"],
    "informes:delete":    ["ingenieria"],
    "datos:read":         ["ingenieria", "produccion"],
    # RRHH
    "operarios:read":     ["rrhh"],
    "operarios:export":   ["rrhh"],
    # Administración técnica (sólo ingeniería lee; sólo propietario escribe)
    "bbdd:read":          ["ingenieria"],
    "conexion:read":      ["ingenieria"],
    "conexion:config":    [],   # sólo propietario (tocar credenciales SQL)
    "email:send":         ["rrhh", "ingenieria", "gerencia"],
    "plantillas:read":    ["ingenieria"],
    "plantillas:edit":    ["ingenieria"],
    # Ajustes — sólo propietario en Mark-III
    "ajustes:manage":     [],
    "auditoria:read":     [],
    "usuarios:manage":    [],
}

def require_permission(permission: str):
    """Factory: devuelve un dependency FastAPI que verifica permiso."""
    async def _check(request: Request):
        user = getattr(request.state, "user", None)
        if user is None:
            # Defensivo: AuthMiddleware debería haber cortado antes.
            raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED)
        if user.role == "propietario":
            return user
        allowed = PERMISSION_MAP.get(permission, [])
        user_depts = {d.code for d in user.departments}
        if not user_depts.intersection(allowed):
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"Permiso requerido: {permission}",
            )
        return user
    return _check
```

**Reglas de estilo**:
- `PERMISSION_MAP` como constante a nivel de módulo.
- `require_permission` es la única entrada pública para routers.
- `propietario` NO aparece como valor en ningún `list` del mapa (research §Anti-Patterns).

**Criterio de hecho**:
- `docker compose exec web python -c "from nexo.services.auth import PERMISSION_MAP; print(len(PERMISSION_MAP))"` imprime ≥ 20.
- Import-check: `from nexo.services.auth import require_permission, PERMISSION_MAP` no lanza.

**Observación**: CONTEXT.md mencionaba 14 routers; el repo tiene además `plantillas.py`. Lo incluimos por coherencia. **Documentar en este plan** que el total real es 15.

**Patrón de retrofit**:

```python
# ANTES
from fastapi import APIRouter
router = APIRouter(prefix="/ciclos", tags=["ciclos"])

# DESPUÉS
from fastapi import APIRouter, Depends
from nexo.services.auth import require_permission

router = APIRouter(
    prefix="/ciclos",
    tags=["ciclos"],
    dependencies=[Depends(require_permission("ciclos:read"))],
)

# Endpoint con permiso estricto adicional:
@router.post("", dependencies=[Depends(require_permission("ciclos:edit"))])
def crear_ciclo(...): ...
```

**Reglas**:
- No tocar la lógica del endpoint — sólo añadir `dependencies=[...]` en el `APIRouter()` y, cuando aplique, en endpoints específicos.
- No importar `require_permission` en módulos que no lo usen.
- Orden sugerido de ejecución (research §Open Question 5): arrancar por `health` (sólo verificar que no tocamos nada), `centro_mando`, `luk4` (readonly). Seguir con `historial`, `informes`, `capacidad`, `datos`. Luego `ciclos`, `recursos`, `operarios`, `plantillas`, `email`. Cerrar con los sensibles: `pipeline`, `bbdd`, `conexion`.
- Tras cada router, correr `docker compose restart web` y hacer un curl al endpoint afectado con cookie de propietario → debe seguir devolviendo 200.

**Criterio de hecho**:
- Los 15 archivos (14 con retrofit + `health` verificado sin tocar) compilan.
- `grep -rn "require_permission" api/routers/ | wc -l` ≥ 20 (cada router con retrofit añade 1-2 líneas).
- `make dev` arranca.
