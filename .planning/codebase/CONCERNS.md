## Tech Debt

### Global Exception Handler Leaks Information (CRITICAL)

**Files:** `api/main.py` (lines 44-57)

**Issue:** The `global_exception_handler` returns the complete Python traceback to the browser in plaintext HTML. This leaks:
- Full stack trace with file paths and function names
- Database connection strings (if error occurs during SQL operations)
- Internal module structure
- Sensitive parameters from function arguments

**Impact:** Any runtime error in `/api/*` endpoints returns 500 with traceback visible in browser. In production (Mark-III), this is a security vulnerability that enables reconnaissance attacks and exposes system internals.

**Fix approach:** 
- Rewrite handler to:
  - Log detailed traceback to server-side logs only (with unique error ID)
  - Return generic error page with error ID to user
  - Include specific error message only if `DEBUG=true` in dev
  - Add audit entry for unhandled exceptions

### Password Field Exposed in GET `/api/conexion/config` (HIGH)

**Files:** `api/routers/conexion.py` (lines 35-49)

**Issue:** The `GET /config` endpoint returns the SQL Server password in plaintext JSON:
```python
"password": cfg.get("password", ""),
```
The comment on line 37 says "con password enmascarada" but the code returns it unmasked.

**Impact:** Any user (authenticated or not in current code) can call `GET /api/conexion/config` and retrieve the SQL Server `sa` password. This is directly readable in browser DevTools, network logs, and proxies.

**Current protection:** Currently there is NO authentication on ANY endpoint, so this is completely open.

**Fix approach:**
- Remove password field from response entirely (users don't need it)
- If password must be displayed for verification: return masked version like `"sa***..."` only
- Require `admin` role permission (Sprint 1 onward)
- Log access to sensitive endpoints in audit trail (Sprint 1 onward)

## Security Considerations

### No Authentication or Authorization (CRITICAL)

**Files:** All routers in `api/routers/` (14 routers, ~3500 LOC)

**Issue:** 
- No decorator checking user identity
- No role-based access control
- No middleware enforcing login
- All endpoints are publicly accessible
- UI doesn't prompt for credentials

**Risk:** Anyone with network access to the LAN server can:
- Read all production data via `/api/luk4/*`, `/api/capacidad/*`, etc.
- Change SQL connection credentials via `PUT /api/conexion/config`
- Delete execution history via `DELETE /api/historial/{id}`
- Trigger expensive operations (pipeline, PDF generation, queries)
- View audit logs (when implemented)

**Fix approach:** Sprint 1 (Identidad + audit) - see MARK_III_PLAN.md lines 98-161

### No Audit Logging (HIGH)

**Files:** All endpoints, no audit mechanism exists

**Issue:** No record of:
- Who accessed what data and when
- Who made changes to configuration
- Who triggered expensive operations
- Failed login attempts

**Risk:** 
- Cannot detect unauthorized access retroactively
- Cannot comply with data protection regulations (GDPR, ISO 27001)
- Cannot troubleshoot "who deleted my execution?"

**Fix approach:** Sprint 1 - implement `nexo.audit_log` append-only table with per-request middleware

### SQL Injection in `/api/bbdd/query` (MEDIUM)

**Files:** `api/routers/bbdd.py` (lines 280-330)

**Issue:** The endpoint accepts arbitrary SQL and applies a whitelist filter:
```python
# Reject keywords: INSERT, UPDATE, DELETE, DROP, CREATE, ALTER, EXEC, xp_
if any(kw in query.upper() for kw in BLOCK_KEYWORDS):
    return {"error": "Query contains blocked keywords"}
```

While it blocks dangerous keywords, this approach is fragile:
- Comments can bypass checks: `SELECT /* DROP */ * FROM table`
- Case variations: `seLeCt`, `SeLeCt` might slip through if uppercase check fails
- Multi-line statements: whitelist-based checks may not catch all injection patterns

**Current protection:**
- Only SELECT queries allowed (reasonable)
- Keywords blocked (incomplete protection)
- No parameterized queries capability offered to users

**Risk:** A sophisticated attacker might craft a query that passes the keyword whitelist but executes malicious operations.

**Fix approach:**
- Use SQL Server's query result limitations instead of keyword blacklist
- Require admin role to use this endpoint (Sprint 1)
- Log all queries and their results to audit trail
- Consider removing this endpoint or making it view-only to a specific analyst role

### Plaintext Password in Memory During SMTP Test (LOW)

**Files:** `api/services/email.py` (lines 20-33)

**Issue:** The SMTP password from config is kept in memory as plaintext during the test connection. While Python will garbage-collect it, it may appear in memory dumps or debug logs.

**Impact:** Minor; only relevant if process is compromised. But following best practices: secrets shouldn't linger in memory longer than necessary.

**Fix approach:** 
- Use context managers to zero out password bytes after use
- Consider external SMTP provider token instead of plaintext password

### SMS Config Located Outside Version Control (MEDIUM)

**Files:** `data/db_config.json` (gitignored)

**Issue:** The only place SMTP configuration lives is in the gitignored `data/db_config.json` file. This means:
- No backup of SMTP config in version control
- No way to know what the current SMTP configuration is without accessing the running server
- If server crashes and disk is lost, SMTP config is gone
- The file currently doesn't have an `smtp` section, so email is broken

**Current status:** `api/services/email.py` reads `cfg.get("smtp", {})` and returns empty dict if missing, causing silent failures.

**Fix approach:**
- Move SMTP configuration to `.env` file with clear `NEXO_SMTP_*` variables
- Update `.env.example` to document SMTP setup
- Validate SMTP configuration at startup (fail fast if misconfigured)

## Performance Bottlenecks

### Matplotlib PDF Generation Blocks Worker (HIGH)

**Files:** 
- `OEE/disponibilidad/main.py` (911 LOC)
- `OEE/rendimiento/main.py` (687 LOC)
- `OEE/calidad/main.py` (599 LOC)
- `OEE/oee_secciones/main.py` (1624 LOC)
- `api/services/pipeline.py` (lines 200-350 where modules are called)

**Issue:** PDF generation using matplotlib is:
- **Synchronous and blocking**: holds the worker thread for minutes
- **CPU-bound**: no async/await, can't use `asyncio`
- **Single worker bottleneck**: with `workers=1` in uvicorn, all other requests block
- **No cancellation**: if a pipeline run takes 5 minutes and user closes browser, it keeps running

**Current mitigation:** None. The endpoint uses Server-Sent Events (SSE) to report progress, but the underlying operations are blocked.

**Impact:** 
- If two users trigger pipeline runs simultaneously, second user waits for first to complete
- Dashboard becomes unresponsive during pipeline execution
- Large runs (20+ resources × 30+ days × 4 modules) can take 10+ minutes

**Estimated timing from MARK_III_PLAN.md (line 300-304):**
- Preflight heuristic: ~10s/resource/day/module
- Example: 10 resources × 30 days × 1 module = 3000s = 50 minutes

**Fix approach:** Sprint 3 or 4 - see MARK_III_PLAN.md:
- Move PDF generation to `asyncio.to_thread()` or worker pool
- Add query timing middleware to measure actual costs
- Implement preflight estimation with amber/red approval flow
- Consider replacing matplotlib with async-compatible library (WeasyPrint, ReportLab) in Mark-IV

### Large DataFrames in Memory (MEDIUM)

**Files:** `OEE/db/connector.py` (lines 400-450, approximate)

**Issue:** The `extraer_datos` function uses `pd.read_sql()` to load entire result sets into memory:
```python
df = pd.read_sql(_SQL_TEMPLATE, con, params=(...))
```

For large date ranges (e.g., 180 days of all machines), this can load millions of rows.

**Risk:**
- Memory spike during extraction
- OOM (out of memory) errors on resource-constrained servers
- GC pauses during processing

**Fix approach:**
- Use cursor iteration instead of `read_sql` for large result sets
- Batch-process rows (e.g., 10K rows at a time)
- Stream results directly to CSV without intermediate DataFrame

### No Query Optimization for Slow Tables (LOW)

**Files:** `OEE/db/connector.py` (lines 150-250, SQL template section)

**Issue:** The main SQL template (lines ~250-350) performs multiple JOINs on large tables (`fmesdtc`, `fprolof`) without indexed hints or query optimization.

**Current state:** The query works but no analysis of execution plans available from this audit.

**Fix approach:** 
- Run SQL Server execution plans on production data
- Add indexes if needed
- Consider materialized views for frequently-accessed aggregations

## Fragile Areas

### SECTION_MAP Duplicated in 4 Locations (MEDIUM)

**Files:**
- `api/database.py` (implied in models)
- `OEE/db/connector.py` (lines 68-77)
- `OEE/utils/excel_import.py` (similar structure)
- `static/js/app.js` (hardcoded literal)

**Issue:** The mapping of machine names to sections (e.g., "luk1" → "LINEAS", "t48" → "TALLADORAS") is defined in multiple places. Changing the mapping requires updating 4 files.

```python
# In connector.py
RESOURCE_SECTION_MAP = {
    "luk1": "LINEAS",
    "luk2": "LINEAS",
    ...
    "t48": "TALLADORAS",
}
```

**Impact:** 
- New machine added to IZARO requires 4 changes
- Easy to miss one location → data routed to wrong section
- Tests only cover Python, so JS mapping can drift

**Fix approach:**
- Centralize in `api/config.py` as a Pydantic settings dictionary
- Load from `.env` or database instead of hardcoded
- Use dependency injection to pass to OEE modules
- Validate against database schema on startup

### TURNOS Helper Functions Duplicated (MEDIUM)

**Files:**
- `api/services/turnos.py` (source of truth, 77 LOC)
- `OEE/oee_secciones/main.py` (lines ~100-200, reimplemented)
- `OEE/rendimiento/main.py` (similar reimplementation)
- `OEE/calidad/main.py` (similar reimplementation)

**Issue:** The shift assignment logic (`determinar_turno`, `calcular_solapamiento_turno`) is defined in `api/services/turnos.py` but then reimplemented in OEE modules, likely to avoid circular imports or because OEE was standalone.

**Risk:** Bug fix in `turnos.py` may not propagate to OEE modules; they calculate shift membership differently.

**Fix approach:**
- Import `turnos.py` functions in OEE modules (may require refactoring imports)
- Or move TURNOS constant to shared location (`OEE/utils/`)
- Add integration test verifying all modules use same shift logic

### SQL Duplicated Across Routers (MEDIUM)

**Files:**
- `api/routers/capacidad.py` (lines 54-129, large SQL query)
- `api/routers/luk4.py` (lines 80-200, 6 large SQL queries)
- `api/routers/centro_mando.py` (lines 45-59, hardcoded 3-part names)
- `api/routers/operarios.py` (4 large SQL queries)
- `OEE/db/connector.py` (main extraction query)

**Issue:** SQL queries are embedded directly in router code, making it:
- Hard to reuse queries across endpoints
- Hard to test queries in isolation
- Easy to accidentally diverge on similar logic
- Impossible to version control query changes separately

**Example:** Both `capacidad.py` and `operarios.py` query `fmesdtc` with similar joins but slightly different aggregations.

**Fix approach:** Sprint 2 (Capa de datos) - see MARK_III_PLAN.md lines 164-235:
- Move all SQL to `.sql` files in `nexo/data/sql/`
- Create repositories with named methods: `capacidad_repo.calcular_p10()`, `operarios_repo.extraer_por_codigo()`
- Use parameterized queries with placeholders

### Connection String Built in 3 Places (LOW)

**Files:**
- `api/database.py` (lines 19-31, `_mssql_creator`)
- `OEE/db/connector.py` (lines 131-160, `_build_connection_string`)
- `scripts/create_oee_db.py` (reimplemented)
- `scripts/migrate_schemas.py` (reimplemented)

**Issue:** The ODBC connection string is built differently in each location, increasing risk of inconsistency.

**Fix approach:**
- Centralize in single utility function
- Reuse across all modules
- Add unit tests for connection string generation

## Known Bugs

### LUK4 Availability Metric Out of Sync with Piece Count (DOCUMENTED)

**Files:** `api/routers/luk4.py`, endpoint `/api/luk4/turno-detail`

**Status:** Documented in `ERRORES.md` (lines 8-62) as NOT OUR BUG.

**Symptom:** In the shift detail panel, `availability_pct` can be low while `piezas_buenas` is high (or vice versa), indicating misalignment.

**Root Cause:** Two separate tables populated independently by the PLC:
- `luk4.estado` → availability percentage
- `luk4.tiempos_ciclo` → piece counts

If the PLC records state changes at different frequencies than counter updates, or if there's a counter reset, the metrics diverge.

**Current Fix:** `luk4.py` reads both tables correctly. The misalignment is a data quality issue in the PLC/MES, not the app.

**Action:** This is a cross-functional issue (MES team needs to sync PLC state sampling frequencies).

### LUK4 Piece Loss at Shift Boundary (FIXED)

**Files:** `api/routers/luk4.py` (fix applied)

**Status:** FIXED in commit unknown before audit date.

**Symptom:** Pieces produced between last reading of shift N and first reading of shift N+1 could fall into a gap and not be counted. Total daily pieces from shifts wouldn't sum to daily total.

**Root Cause:** Endpoint calculated `first_turno` as the first reading *within* the shift, leaving a gap before the shift start.

**Fix Applied:** Changed to last reading *before* shift start, with fallback for first historical shift.

**Current Status:** ✓ Working as intended.

## Scaling Limits

### Single ODBC Driver Instance (MEDIUM)

**Files:** `api/database.py` (lines 33-40, SQLAlchemy pool)

**Issue:** The pool is configured with:
```python
pool_size=5,
max_overflow=10,  # Total: 15 concurrent connections
pool_recycle=3600,
```

For 15 concurrent database connections at max capacity:
- Database growth on SQL Server (each connection uses memory)
- Connection limits on driver
- Network saturation if all connections are active

**Estimated limit:** With `workers=2` in uvicorn (2 worker processes × 7 threads per worker = 14 workers), all 15 pool connections could be saturated under moderate load.

**Fix approach:**
- Benchmark actual connection usage under production load
- Monitor connection pool statistics (Spring 4)
- Consider connection pooling at OS level (PgBouncer for Postgres in Sprint 1)
- Increase pool if needed, with corresponding SQL Server configuration

### Matplotlib PDF Generation on Single Worker (MEDIUM)

**Files:** `api/services/pipeline.py`, `OEE/**/main.py`

**Issue:** With `workers=1` (default for small deployments), a single 10-minute PDF generation blocks all other requests. Even with `workers=2`, only one request can run PDF pipelines in parallel.

**Estimated capacity:** 
- 1 full pipeline run per worker per 10 minutes (rough estimate)
- With 2 workers: 2 pipelines per 10 minutes max

If 3 users want to run reports simultaneously, one waits 10+ minutes.

**Fix approach:** Sprint 3 - move PDF generation to thread pool

### Postgres Connection Not Used Today (LOW)

**Files:** `docker-compose.yml` (postgres service), `.env` (PG_USER, PG_PASSWORD, PG_DB)

**Issue:** Postgres is defined in compose but never connected to:
- `settings.database_url` is constructed for mssql
- Engine ignores the Postgres URL
- No Alembic migrations
- No code paths read from Postgres

**Impact:** Wastes container resources (memory, disk) and confuses developers ("why is Postgres here if we don't use it?").

**Current status:** Postgres will become important in Sprint 1 for `nexo.users`, `nexo.audit_log`, `nexo.roles`, etc.

**Mitigation for Mark-II:** Postgres is overhead today but will be needed for Mark-III. No immediate action needed; plan to populate it in Sprint 1.

## Missing Critical Features

### No API Rate Limiting (HIGH)

**Files:** All routers, no middleware for rate limiting

**Issue:** 
- No protection against brute force login (will be added Sprint 1)
- No protection against DoS/resource exhaustion
- Users can spam expensive endpoints (pipeline, capacity queries) repeatedly

**Risk:**
- Single malicious user can lock up the server
- Expensive queries can be triggered 100 times per second

**Fix approach:** 
- Add rate limiting middleware per user/IP (Sprint 1)
- At minimum: 10 req/sec per user on expensive endpoints

### Missing Input Validation (HIGH)

**Files:** Several routers accept date ranges, resource lists, etc. without explicit validation

**Example issues:**
- `/api/capacidad?desde=...&hasta=...` doesn't validate date format
- `/api/historial/regenerar?fecha=...` assumes valid date
- `/api/pipeline/run` accepts `fecha_inicio`, `fecha_fin` without type checking

**Risk:** Invalid input could cause silent failures or unexpected behavior

**Fix approach:**
- Use Pydantic `BaseModel` for all endpoint parameters
- Add validators for date format, numeric ranges
- Return 422 Unprocessable Entity with clear error messages

### No Health Check for MES Connectivity (MEDIUM)

**Files:** `api/routers/health.py` (32 LOC)

**Issue:** The `/api/health` endpoint doesn't test actual connectivity to the MES database (`dbizaro`). It only tests the local engine or makes a simple ping.

**Impact:** The UI shows "Connected to MES" but the actual query endpoint could fail if MES is down, causing cryptic errors downstream.

**Fix approach:**
- `health.py` should test a simple read query against `dbizaro`
- Return status like `{"web": "ok", "ecs_mobility": "ok", "dbizaro": "warn", "reason": "timeout"}`

## Configuration and Versioning Issues

### No Explicit Database Versioning (MEDIUM)

**Files:** `scripts/create_oee_db.py`, `scripts/migrate_schemas.py`, `scripts/extract_2025.py`

**Issue:** Database schema is initialized via idempotent Python scripts, not declarative migrations. There's no `schema_version` table to track which migrations have run.

**Current state:**
- `create_oee_db.py` creates tables if they don't exist
- `migrate_schemas.py` moves tables between schemas
- `extract_2025.py` imports one-time data

**Risk:** 
- If a script partially fails, it's unclear which changes applied
- Deploying to a new environment requires running scripts in correct order (not automated)
- No way to roll back a migration

**Fix approach:** Sprint 2 (Capa de datos) includes `schema_guard.py` to validate schema at startup

### Inconsistent Naming Conventions (MEDIUM)

**Files:** Throughout codebase

**Issue:**
- Table names: `cfg.ciclos`, `oee.ejecuciones`, `luk4.estado` (mixed singular/plural)
- Column names: `fecha_inicio` vs `start_date`, `ct_code` vs `codigo`, `maquina` vs `machine`
- Function names: `convertir_raw_a_metricas` (Spanish) vs `test_connexion` (English with typo)
- Variable names: `cfg`, `config`, `settings` used interchangeably

**Risk:**
- Confusing for new developers
- Harder to search for references
- SQL queries fragile to column renames

**Fix approach:**
- Establish naming standard in `CLAUDE.md` (Sprint 0)
- Gradually standardize in refactors (Sprints 2-4)
